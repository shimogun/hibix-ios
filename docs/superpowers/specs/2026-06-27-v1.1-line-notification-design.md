# Hibix iOS v1.1 LINE通知 — 設計書（タスク#10）

- 日付: 2026-06-27
- 対象: `hibix-ios`（iOSアプリ）
- 前提: backend は本番稼働済み（C案: email/LINE 同列 contact_type、PR #5 マージ・deploy 済）
- 関連: `~/.company/secretary/notes/2026-06-26-hibix-line-v1.1-handoff.md` / 同 `-backend-impl-result.md`
- 真実の契約: backend `src/routes/contacts.ts` / `src/routes/settings.ts` の Zod + 本番稼働コード

## 0. 背景と確定事項

- 連絡先は `email` / `line` の**同列の種別**（`contact_type`）。`email_*` は nullable（line型はNULL）。
- 通知モード（gentle/daily）では **email 型連絡先が最低1件必須**（サーバー M-01: 不足時 400 `EMAIL_CONTACT_REQUIRED`）。
- LINE 連携は**コード送信方式**。ユーザーはLINE IDを入力しない。
- **重要発見**: iOS は現状 contacts/settings をサーバー同期していない（`.contacts`/`.settings` 定義済みだが呼び出しゼロ）。LINE 連携はサーバー contact UUID 前提のため、**同期実装が前提条件**。

### 確定した設計判断（オーナー承認 2026-06-27）
1. **同期トリガ**: ローカル変更の都度に即同期（PUT/PATCH）。
2. **失敗時UX**: 通常はサイレント＋次回再同期（checkin同様）。LINE連携導線のみ画面内でエラー明示。
3. **連携状態の保持**: `server_id` ＋ `line_link_status` をローカルDBに保持（オフラインでもバッジ表示）。

## 1. アーキテクチャ

既存 `CheckinService`（APIClient＋Repository を束ねるサービス層）パターンを踏襲。ローカル `Int64 id` は維持し、サーバーUUIDは `server_id` 列で別管理（最小侵襲）。

```
[EmergencyContactEdit/ModeSwitch UI]
        │ ローカル変更（add/update/delete/モード変更）
        ▼
[ContactsSyncService]  ──PUT /api/contacts / PATCH /api/settings──┐
   ・ローカル全件→PUT本体生成                                      │
   ・200レスポンスの id を sort_order順 index で                   ├─> [APIClient]（App Attest 4ヘッダ自動付与）
     ローカルrowへ server_id 書き戻し                              │
   ・失敗サイレント＋resyncOnLaunch                                │
                                                                  │
[LineLinkView/ViewModel] ─> [LineLinkService] ──issue-code/status─┘
   ・server_id 必須（無ければ先に syncContacts(throws)）
   ・line_link_status をローカル更新
```

## 2. データ層（GRDB migration v3）

`Migrations.swift` に v3 を1登録追加（純加算・後方互換）:

```
v3_emergency_contacts_add_line_sync:
  ALTER TABLE emergency_contacts ADD COLUMN server_id TEXT;
  ALTER TABLE emergency_contacts ADD COLUMN line_link_status TEXT NOT NULL DEFAULT 'unlinked';
```

- `EmergencyContact` モデル: `serverID: String?` / `lineLinkStatus: LineLinkStatus` を追加。
- 新 enum `LineLinkStatus: String { unlinked, pending, linked }`（`fromStoredValue` で未知値→unlinked）。
- `EmergencyContactsRepository`:
  - `list()` / `decode` / `add` / `update` を新列対応（add は server_id=nil, status='unlinked' で挿入）。
  - 追加メソッド: `updateServerMapping(_ pairs: [(localID: Int64, serverID: String)])`、`updateLineLinkStatus(localID: Int64, status: LineLinkStatus)`。
  - `update` は既存の server_id / line_link_status を**保持**（email/label/contact_type のみ更新）。種別を line→email 等に変えた場合の line_link_status 扱いは「種別変更時は unlinked へリセット」。

## 3. ネットワーク層（契約修正）

`APIEndpoint.swift` の既存 body/response が旧契約（`kind`・`email`必須・contact_typeなし）なので本番契約へ修正。

### リクエスト（PUT /api/contacts）
```swift
struct ContactInputBody: Encodable {
    let id: String?            // server uuid。省略=新規（JSON では nil 時キー省略）
    let contact_type: String   // "email" | "line"（旧 kind から改名）
    let email: String?         // email型のみ。line型は nil
    let label: String?
}
struct ContactsPutBody: Encodable { let contacts: [ContactInputBody] }
```
- `id == nil` のときは JSON にキーを出さない（`encodeIfPresent`）。

### レスポンス（PUT /api/contacts）
```swift
struct ContactOutput: Decodable {
    let id: String
    let contact_type: String   // 追加
    let label: String?
}
```
- 並び順は sort_order 昇順＝リクエスト順。**index 対応**でローカルへ server_id 書き戻し。

### PATCH /api/settings
- 既存 `SettingsPatchBody{watch_days:Int?, watch_mode:String?}` / `SettingsResponse{watch_days, watch_mode, is_pro}` をそのまま使用（呼び出しを実装するのみ）。

### 新エンドポイント
```swift
case lineIssueCode(serverContactID: String)  // POST /api/contacts/{id}/line/issue-code, body無し, requiresAttest=true
case lineStatus(serverContactID: String)     // GET  /api/contacts/{id}/line/status,    body無し, requiresAttest=要確認(実装時にbackend確認)
```
- パス組み立て: `"/api/contacts/\(serverContactID)/line/issue-code"`。
- **epoch秒はInt型でデコード**（他APIのISO8601 Dateと異なる。`JSONDecoder.dateDecodingStrategy=.iso8601` では壊れるため Int で受けて変換）:
```swift
struct LineIssueCodeResponse: Decodable { let code: String; let expires_at: Int; let add_friend_url: String? }
struct LineStatusResponse: Decodable { let status: String; let code_expires_at: Int? }
```
- 実装着手時に backend `contacts.ts` の status/issue-code ルートの authMiddleware / app-attest 適用を実コードで最終確認し、`requiresAttest` を確定する。

## 4. サービス層

### ContactsSyncService（@MainActor）
- 依存: `APIClient` / `EmergencyContactsRepository` / `SettingsRepository` / `AppAttestClient`（attest未登録なら read-only スキップ＝checkin同様）/ `DeletionPendingCoordinator?`。
- `syncContacts() async`（サイレント）/ `syncContactsThrowing() async throws`（LINEフロー用）:
  1. ローカル全件を sort_order 順で取得。
  2. `ContactsPutBody` 生成（server_id 既知は id に載せ、line型は email=nil）。
  3. PUT 実行。200 の `contacts[]` を index 対応でローカル row に `server_id` 書き戻し。
  4. 失敗は os_log のみ（throwing版は再送出）。`DELETION_PENDING` は coordinator へ。
- `syncSettings(watchMode:watchDays:) async throws`: PATCH 実行。`EMAIL_CONTACT_REQUIRED` を呼び出し元へ伝播。
- `resyncOnLaunch() async`: 起動時に best-effort で `syncContacts()`（checkin の resync と同じ思想で `RootView`/起動導線から呼ぶ）。

### LineLinkService（@MainActor）
- 依存: `APIClient` / `EmergencyContactsRepository` / `ContactsSyncService`。
- `func issueCode(localContactID: Int64) async throws -> LineIssueCodeResult`:
  - 対象ローカル連絡先の `server_id` が無ければ `contactsSync.syncContactsThrowing()` で取得 → 再取得。
  - POST issue-code → `code / expiresAt(Date) / addFriendURL`。ローカル status を `pending` に更新。
- `func fetchStatus(serverContactID: String, localContactID: Int64) async throws -> LineLinkStatus`:
  - GET status → ローカル status を反映（linked/pending/unlinked）。

## 5. UI層

### EmergencyContactEditView
- `isDeliveredInV01` の "coming soon" 注記（現 L56-67）を**撤去**。
- `contactType == .line` のとき「LINE連携」セクションを表示:
  - 現在の `lineLinkStatus` バッジ（未連携 / 連携待ち / 連携済み）。
  - 「LINEで連携する」ボタン → （新規未保存なら save→sync→server_id取得後）`LineLinkView` を提示。
  - 既存 linked の場合は「連携済み」表示（再連携導線は任意・最小実装では非表示）。

### LineLinkView / LineLinkViewModel（新規）
- 表示時に `issueCode` 実行 → 6桁コードを大字表示。
- 「友だち追加」ボタン（`add_friend_url` を `Link`/openURL）。
- `ShareLink` でコード＋友だち追加URL＋簡単な手順文を共有。
- status を約4秒間隔ポーリング（画面表示中のみ。`linked` で停止＆成功表示、ローカル status 更新）。
- コード失効（`expiresAt` 経過）で「コードを再発行」ボタン。
- エラーは画面内に表示（Q1: LINE導線のみ明示）。
- 文言は薬機法NG表現を避ける（「見守り」等の既存トーン踏襲）。

### ModeSwitchViewModel.select()
- gentle/daily 切替前に**ローカル email 連絡先 ≥ 1 を先回りチェック**（0 なら誘導アラートを出し切替しない＝サーバー M-01 を未然回避）。
- 切替成功時に `syncSettings(watchMode:watchDays:)`。`EMAIL_CONTACT_REQUIRED` が返ったら（競合等）ローカルを元に戻しアラート。

### 削除/種別変更ガード（M-01 先回り 2箇所目）
- gentle/daily 中に**最後の email 連絡先**を削除/別種別へ変更しようとしたらブロック（`EmergencyContactsViewModel.delete` / `EmergencyContactEditViewModel.delete`・種別変更保存）。
