## 3. 機能設計

### 3.1 コアフロー(デイリー)

```
朝/夜の通知
   ↓
アプリ起動 → 気分タップ(色グラデーション選択 + 任意メモ)
   ↓
即時保存(ローカル)
   ├─→ 自分のピクセルカレンダー・グラフに蓄積
   └─→ チェックイン成功(タイマーリセット)
```

### 3.2 異常時フロー(自動)

```
n日タップなし(ユーザー設定: 1〜7日)
   ↓
本人にプッシュ通知でリマインダー(1回)
   ↓
それでも反応なし
   ↓
緊急連絡先へメール通知
   「○○さんが◯日間記録していません」
   ＋ Web招待リンク(最終チェックイン日表示)
```

### 3.3 3つの見守りモード(差別化の核)

| モード | 動作 | 想定ユーザー |
|---|---|---|
| 🌱 **おひとりさまモード** | 完全プライベート、見守り通知なし | メンタル日記目的の単独利用 |
| 🤝 **ゆるつながりモード** | 異常時のみ緊急連絡先へ通知 | 一人暮らし×離れた家族 |
| 💚 **まいにち共有モード** | 毎日「○○さん、今日は😊でした」を相手に通知 | 親子・恋人・闘病サポート |

ユーザーが設定で自由に切り替え可能。

### 3.4 UI核: ピクセル/色グラデーション

GitHubコントリビューションカレンダー風 + Year in Pixels風。

```
■■■■■■■  ← 1年分のカレンダー
■■□■■■■    各セルが日別気分の色
■■■■□■■    タップで日別詳細・メモ閲覧
□■■■■■■    色は5〜7段階のグラデーション
```

**v0.7 追加: 「直近1年」の解釈確定**

無料層の「直近1年」表示は、**解釈A: 直近365日ローリングウィンドウ**で確定。

```
今日が 2026-12-31 → 表示範囲: 2026-01-01 〜 2026-12-31
今日が 2027-01-01 → 表示範囲: 2026-01-02 〜 2027-01-01
```

実装ロジック: `WHERE date >= today - 365 days`

GitHubコントリビューションカレンダー(草グラフ)と同じ挙動。日が進むたびに最も古い1日分が表示から外れる。データは消えず、有料層では全期間スクロール可能。

差別化:
- 競合(Daylio/MOODA/muute)と視覚的に明確に違う
- スクショだけで世界観が伝わる
- ASOキーワード: 「年in pixels 日本語」「気分記録 ピクセル」

## 4. データ保護モデル

### 4.1 設計原則

**「心の中身は端末から出さない、安否のpingだけクラウド経由」**

| データ種別 | 保存場所 | 理由 |
|---|---|---|
| 日記本文・気分ピクセル・履歴・グラフ | **端末ローカル100%** | 一番センシティブな"心の中身"は外に出さない |
| チェックインのタイムスタンプのみ(例: `2026-05-15 09:32 ping`) | 最小サーバー | 通知発火に必要な最小情報。**内容は一切含まない** |
| 緊急連絡先メールアドレス(暗号化) | 最小サーバー | 通知送信先 |

### 4.2 マーケコピー用説明

> 「あなたの記録は1mmも外に出ません。
> サーバーが知っているのは"今日ログインした"という事実だけ。
> 何を書いたか、どんな気分だったかは、永遠にあなたのiPhoneの中だけ。」

### 5.1 技術スタック(v0.7 で正規化)

**【重要】**: v0.6 添付版では §5.1 がVercel記述のまま残されていたが、これは反映漏れ。本v0.7 で v0.6 確定事項である Cloudflare Workers + D1 に正規化する。

| 領域 | 採用技術 | バージョン |
|---|---|---|
| iOSアプリ | **SwiftUI(Native)** | iOS 17.0+ / Swift 5.10+ / Xcode 16+ |
| ローカルDB | **GRDB.swift** | 6.29.0+ |
| Keychain | KeychainAccess(iCloud同期有効) | 4.2.2+ |
| 認証 | **匿名UUID + iOS Keychain** | iCloud Keychain で機種変対応 |
| 課金 | **StoreKit 2** | iOS 17 SDK |
| サーバー | **Cloudflare Workers** | Wrangler 3.x / TypeScript 5.x |
| サーバーDB | **Cloudflare D1(SQLite)** | — |
| ORM | **Drizzle ORM(drizzle-orm/d1 アダプター)** | 0.30+ |
| Cron | **Cloudflare Cron Triggers** | 15分間隔 |
| メール送信 | **Resend** | — |
| 暗号化 | iOS Keychain(端末側)+ AES-256-GCM(サーバー側) | — |
| シークレット管理 | **wrangler secret** | — |

**Apple Sign-In は v0.1 では採用しない**(匿名UUID + iCloud Keychain で機種変対応するため)。

**Cloudflare採用の根拠(v0.6 確定事項)**:
- 商用利用無料スタート可(Vercel Pro は $20/月必須)
- DDoS / WAF / Bot Fight Mode が無料枠標準装備
- 10万DAUまで無料枠内で運用可能
- Web標準API準拠で将来移植性高(ベンダーロックイン軽)
- D1 は SQLite互換、GRDB(端末側)とスキーマ表現を揃えやすい

### 5.2 セキュリティ運用ルール

| 項目 | ルール |
|---|---|
| HTTPS | 全通信HTTPS強制。HSTS preload申請 |
| 入口認証 | **App Attest**(`DCAppAttestService`)の assertion をミドルウェアで検証。全 mutating API で必須(`/api/health` および attest 初期化エンドポイントを除く)。詳細は PRD §8.1 / §10.7 |
| 課金検証 | **StoreKit JWS をサーバー側で Apple Root CA G3 検証**。`is_pro` はサーバー派生値で、クライアント申告は受け付けない。詳細は PRD §8.9 |
| CORS | `Access-Control-Allow-Origin: null`(iOS ネイティブからのアクセスのみ想定。WebView 等の Origin が来たら拒否) |
| Cloudflare 補強 | Bot Fight Mode 有効化、Rate Limiting Rules で IP あたり 60 req/min ハードキャップ |
| レート制限 | 1ユーザーあたり checkin 1日10回まで(D1 上で条件付き UPDATE による atomic 更新) |
| 監査ログ | 緊急連絡先メール送信は全件ログ。30日で自動削除 |
| 依存パッケージ | `pnpm audit` を CI で必須化、HIGH以上は即修正 |
| シークレットスキャン | Gitleaks を pre-commit hook と CI で実行 |
| データ削除権 | ユーザーリクエストで48時間以内に全データ削除。48h 経過前は `POST /api/account/cancel-deletion` で取り消し可 |
| バックアップ | D1の日次自動バックアップ。30日保持 |

### 5.3 将来コスト青天井防止原則

**原則**: 「ユーザー数の増加が線形にコストへ反映される経路を、設計段階で全部塞ぐ」

| コスト源 | 暴騰経路 | 設計上の防止策 |
|---|---|---|
| サーバー実行 | リクエスト数 × 単価 | Cloudflare Workers無料枠 10万req/日 内。10万DAUまで無料 |
| DB | 行数 × R/W | D1無料枠 5GB / 500万行read/日 |
| メール送信 | 通知数 × 単価 | Resend無料枠 3,000通/月。異常時のみ送信 |
| プッシュ通知 | - | APNs使用(サードパーティ通知サービス不採用) |
| AI機能(v0.2) | API呼び出し × トークン | 月額¥480契約者のみ / 1ユーザー月20回まで |
| 画像/ストレージ | - | ローカル保存原則によりサーバー側ストレージ不要 |

**ハードリミット仕様(設計確定事項)**:

| 項目 | 上限 | 理由 |
|---|---|---|
| AI機能呼び出し(v0.2) | 1ユーザー月20回 | コスト青天井防止 |
| 緊急連絡先メール送信 | 同一ユーザーに24時間1回まで | スパム化防止 |
| 緊急連絡先登録数 | 1ユーザー最大3件 | メール送信先暴発防止 |
| ping受付 | 1ユーザー1日10回まで | 誤実装・攻撃の防止 |

### 15.2 修正詳細

#### §2 技術スタック確定

| 項目 | v2.0(現状) | v2.1(修正後) |
|---|---|---|
| サーバー | Vercel + Next.js (App Router) | **Cloudflare Workers** |
| ランタイム | Node.js 24 LTS | **V8 Isolate (Workers Runtime)** |
| フレームワーク | Next.js 15.x | **Hono または素のWorkers fetch handler** |
| サーバーDB | Neon Postgres | **Cloudflare D1 (SQLite)** |
| ORM | Drizzle ORM | **Drizzle ORM (drizzle-orm/d1 アダプター)** |
| Cron | Vercel Cron(15分間隔) | **Cloudflare Cron Triggers(15分間隔)** |
| シークレット | Vercel環境変数 | **wrangler secret** |
| デプロイ | `vercel deploy` | **`wrangler deploy`** |

#### §3 プロジェクト構造

`/backend` 配下を Cloudflare Workers 標準構成に書き換え:

```
backend/
├── wrangler.toml          ← Workers設定
├── src/
│   ├── index.ts           ← fetch handler エントリポイント
│   ├── routes/
│   │   ├── ping.ts
│   │   ├── user.ts
│   │   └── delete.ts
│   ├── db/
│   │   ├── schema.ts      ← Drizzle スキーマ(SQLite)
│   │   └── client.ts
│   ├── lib/
│   │   ├── crypto.ts      ← AES-256-GCM
│   │   ├── email.ts       ← Resend クライアント
│   │   └── auth.ts
│   └── cron/
│       └── checkin-monitor.ts
├── migrations/
│   └── 0001_initial.sql
└── package.json
```

(Next.js の `app/` ディレクトリ構造は不要)

#### §4 データモデル

Drizzleスキーマを Postgres → SQLite 方言に書き換え:

- `pgTable` → `sqliteTable`
- `serial` / `integer` の扱いを SQLite 仕様に合わせる
- `timestamp` → `integer` (Unix epoch) または `text` (ISO 8601)
- `boolean` → `integer` (0/1)
- 外部キー制約の宣言方式調整

スキーマ自体の構造(テーブル数・カラム名・関係性)は変更不要。

#### §8 API契約

エンドポイント仕様(URL / メソッド / リクエスト/レスポンスJSON / バリデーション)は**そのまま維持**。

ランタイム実装レベルの記述のみ修正:
- `req: NextRequest` → Workers の `Request`
- `NextResponse.json()` → `Response.json()` または Hono の `c.json()`
- ミドルウェア記述方式

#### §13 実装順序(Sprint 1)

Sprint 1 の「バックエンドセットアップ」手順を修正:

| 旧手順(Vercel) | 新手順(Cloudflare) |
|---|---|
| `npx create-next-app` | `npm create cloudflare@latest` で Workers プロジェクト作成 |
| Vercel CLI ログイン | `wrangler login` |
| Neon Postgres プロビジョン | `wrangler d1 create hibix-db` |
| `.env.local` 設定 | `wrangler secret put` でシークレット登録 |
| `vercel dev` | `wrangler dev` |

Sprint 2 以降の機能実装手順は変更不要。
